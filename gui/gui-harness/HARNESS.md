> **DEPRECATED:** The external reference card concept has been removed. References to external cards in this document are no longer valid.

## Playbook

### Group 1: Canvas renders module graph from filesystem

[init] A valid ArchUI project exists with a root module containing at least two submodules, each with valid README.md frontmatter. The GUI application is launched and pointed at the project root.

[action] Open the GUI and navigate to the root canvas.
[eval] Each direct submodule of the root is rendered as a visible node on the canvas, displaying the correct name and description from their README.md frontmatter.

[action] Inspect the edges on the canvas.
[eval] Directed edges are drawn for every link declared in the visible nodes' README.md frontmatter, matching the link targets visible on the canvas.

[end] Close the GUI. The filesystem is unchanged.

### Group 2: Human file edits are reflected live in the GUI

[init] The GUI is open with the root canvas displayed. A submodule node is visible with a known name and description.

[action] In an external text editor, edit the submodule's README.md to change the description field. Save the file.
[eval] The GUI detects the file change and re-renders the corresponding node with the updated description, without requiring a GUI restart.

[action] In an external text editor, add a new subfolder with a valid README.md under the active module.
[eval] The GUI detects the new module folder and adds a new node to the canvas for the new submodule.

[end] Revert all external edits to their original content. Confirm the canvas returns to its prior state.

### Group 3: LLM sync is on-demand only

[init] The GUI is open. The user has made several edits via the canvas (e.g., renamed a module, added a link). These changes have been written to disk as immediate writes.

[action] Verify that no LLM sync has been triggered automatically after the edits.
[eval] No sync indicator is active; no AI-generated patches have been applied. The filesystem reflects only the direct edits.

[action] The user explicitly clicks the LLM sync button in the GUI chrome.
[eval] The sync process starts: a spinner or progress indicator appears. After completion, affected files are updated with propagated changes. The sync indicator returns to idle state.

[end] Revert the project to its pre-edit git state using version control. Confirm the canvas reflects the reverted state.

### Group 4: Node positions are stored as layout hints, not module hierarchy

[init] A canvas is open with multiple module nodes visible. The filesystem module hierarchy is confirmed by inspecting folder structure.

[action] Drag a module node to a new position on the canvas.
[eval] The node moves visually. The folder structure on disk is unchanged. The new position is written to .archui/layout.yaml under the active module's entry.

[action] Reload the GUI or re-open the canvas for the same module.
[eval] The node appears at the position saved in .archui/layout.yaml, not at an auto-generated default position.

[end] Delete the layout entry for the moved node from .archui/layout.yaml. Confirm the node falls back to auto-layout on next canvas load.

### Group 5: Screenshot verification confirms design consistency

[init] The GUI is running on at least one platform (Web). A canvas is open showing a primary card with submodule ports, at least two external reference cards, and both direct edges and port edges. The Figma MCP is connected and the canonical design file is accessible.

[action] Trigger a full screenshot capture of the current canvas.
[eval] The screenshot captures all visible elements: primary card (name, UUID, description, module-level handles, port section), external reference cards (name, UUID), direct edges with arrows, and port edges with arrows. No elements are clipped or missing.

[action] Export the corresponding Figma frame at matching resolution via MCP and compare pixel-by-pixel against the screenshot.
[eval] The pixel diff is within the configured tolerance threshold (default 0.1%). Card dimensions, border-radius, colors, typography, spacing, shadows, and edge styles all match the Figma spec.

[action] Modify a design-affecting spec (e.g., change primary card background color token), trigger the Design Update Trigger pipeline, and re-capture the screenshot.
[eval] The new screenshot reflects the spec change. The Figma frame has been updated by Phase 1. The platform code has been regenerated by Phase 2. The pixel diff between the new screenshot and the updated Figma export is within tolerance.

[end] Revert the spec change and re-run the pipeline to restore original state. Confirm screenshots match the original Figma baseline.
