## 概述

竞品对比回答"市场上已有这么多方案，为什么是我们？"。目标是通过客观数据展示差异化，而非贬低竞争对手。

## 差异化叙事

### 我们的根本不同之处

**ArchUI 是唯一从第一天就为 agentic thinking 设计的企业级开发知识基础设施。**

其他方案都是"先为人类设计，再让 AI 适配"，而我们是"先为 agent 设计，人类通过 GUI 操作"。这个设计哲学的差异体现在：

1. **UUID 链接系统**：其他方案用文件路径或手动超链接，重命名就断链；我们用 UUID，链接永远稳定
2. **渐进式披露**：其他方案是"一次性加载全部内容"；我们是"description → README → resources"三层展开，适配有限的 context window
3. **Schema 验证**：其他方案是"自由文本，agent 自己解析"；我们是"强制结构，agent 行为可预测"
4. **关系语义**：其他方案是"链接就是链接"；我们的链接带语义（depends-on、implements、extends），agent 可以推理

### 为什么竞品无法轻易复制

**短期护城河（6-12 个月）**：

1. **Schema 设计的先发优势**：我们定义了"什么是好的 agent-native 文档结构"，这个标准一旦被企业采用，迁移成本高
2. **GUI 的完成度**：节点式画布 + 实时同步 + 跨平台（Web/Desktop/Mobile），需要大量工程投入
3. **企业客户锁定**：一旦企业将文档结构化到 ArchUI，迁移到其他系统需要重新组织所有知识

**长期护城河（12+ 个月）**：

1. **网络效应**：越多企业使用，agent 的训练数据越好，我们可以提供更智能的文档维护建议
2. **生态系统**：第三方 agent、插件、集成（CI/CD、IDE、监控系统）
3. **数据优势**：企业级文档的结构化数据可以训练更好的领域特定 agent

### 竞品定位分析

| 竞品 | 定位 | 我们的差异 |
|------|------|-----------|
| **Confluence** | 企业协作文档平台 | 我们是"代码级"的文档系统，不是"产品级"的协作工具；我们为 agent 设计，不是为人类会议记录设计 |
| **Obsidian** | 个人知识管理工具 | 我们是"团队级"的基础设施，不是"个人级"的笔记工具；我们有 schema 验证和多人协作，不是自由文本 |
| **Backstage** | 开发者门户 / Service Catalog | 我们是"知识层"的组织，不是"服务层"的 catalog；我们管理架构文档和依赖关系，不是服务注册和监控 |
| **Swimm** | 代码文档工具 | 我们是"架构级"的文档，不是"代码级"的注释；我们关注跨仓库的系统理解，不是单个函数的解释 |

### 核心差异总结

**Confluence**：他们解决"人与人的协作"，我们解决"人与 agent 的协作"
**Obsidian**：他们是个人笔记，我们是团队基础设施
**Backstage**：他们是服务目录，我们是知识图谱
**Swimm**：他们在代码里写文档，我们让文档成为 canonical

## 为什么现在是时机

1. **Agent 能力突破**：2025-2026 年，AI agent 从"代码补全"进化到"多步骤任务执行"，但知识基础设施还停留在"人类阅读"时代
2. **Context window 瓶颈**：即使 Claude 有 200K context，企业级项目（50+ 仓库）仍然无法一次性加载，需要结构化的知识组织
3. **文档熵增加速**：代码由 agent 生成后，文档更新的负担反而更重（因为变化更快），传统文档系统无法跟上
4. **企业 AI 采用率上升**：越来越多企业开始用 AI 辅助开发，但发现"AI 不理解我们的系统架构"成为瓶颈
