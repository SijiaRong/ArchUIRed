## 概述

问题陈述是整个项目叙事的起点。好的问题陈述让评委在 30 秒内理解"为什么这个问题值得解决"。

## 结构模板

### 目标用户

**主要用户：架构师 / Tech Lead**

负责 10+ 个仓库的技术决策者，需要理解跨仓库的依赖关系、向团队传递架构知识、评估重构的影响范围。他们的工作核心是"在复杂度中建立秩序"。

**次要用户：项目主管**

当架构文档清晰后，项目主管自然获益：新人 onboarding 更快、团队对齐成本更低、知识传承更可靠。

### 现有痛点

**当前解决方案及其缺陷：**

1. **Confluence / Notion**：文档和代码分离，更新不及时，AI agent 难以导航结构化信息
2. **README.md in repo**：和代码在一起但跨仓库关系无法表达，缺乏结构化约束
3. **Obsidian + AI 插件**：个人工具，无多人协作能力，无 schema 验证
4. **RAG（向量检索）**：每次重新检索，无知识积累，上下文碎片化

**根本缺陷**：这些工具都是为"人类阅读"设计的，AI agent 只能勉强适配。在 agentic engineering 时代，知识基础设施必须从第一天就为 agent 设计。

### 量化数据

- **理解新系统的时间成本**：架构师平均需要 2-4 周才能理解一个陌生的大型系统
- **新人 onboarding 周期**：平均 3-6 个月才能独立工作，这期间几乎没有产出
- **文档滞后**：架构文档平均滞后实际代码 3-6 个月
- **协同成本**：跨团队对齐每周消耗 5-10 小时的会议时间
- **市场规模**：所有拥有 10+ 仓库的企业（中型到大型科技公司）都面临这个问题

### 情感共鸣

**场景 1：架构师的噩梦**

一个电商平台有 60 个微服务仓库，每个仓库都有自己的 README 和 API 文档。当架构师想要理解"订单服务和支付服务之间的依赖关系"时，他需要：

1. 手动打开 10+ 个仓库的文档
2. 在 Slack、Notion、Confluence 里搜索历史讨论
3. 问三个不同的 Tech Lead"当时为什么这么设计"
4. 最后发现关键的设计决策只存在于一个已经离职的工程师的脑子里

**场景 2：新人的迷宫**

一个新加入的高级工程师，在大厂有 10 年经验，但面对公司的代码库集群完全无从下手。README 要么过时，要么过于简略。他花了 3 个月才理解系统全貌，而这 3 个月里他几乎没有产出。更糟糕的是，当他终于理解了系统，却发现自己理解的已经是"3 个月前的架构"。

**场景 3：Agent 的困境**

一个 AI agent 被要求"优化支付服务的性能"。它读取了支付服务的代码，但不知道：
- 这个服务依赖哪些其他服务？
- 修改这个服务会影响哪些下游系统？
- 历史上为什么选择了这个架构？
- 有哪些已知的技术债务？

结果：agent 做了一个"局部最优"的优化，但破坏了全局的架构一致性。第二天上线后，三个下游服务同时出现故障。

## 一句话问题陈述

**架构师和 Tech Lead 在管理 10+ 个仓库的企业级项目中，面临跨仓库依赖关系不清、架构知识传承困难、AI agent 无法可靠导航的痛点，导致理解系统需要数周、新人 onboarding 需要数月、架构文档持续腐烂，而现有方案（Confluence、README、Obsidian、RAG）无法解决，因为它们都是为人类阅读设计的，缺乏 agent-native 的结构化约束和跨仓库链接能力。**
